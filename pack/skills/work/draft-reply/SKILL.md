## When to Use
- 用户说"帮我回一下 / 帮我写封邮件 / 把这段写正式点 / 帮我润色 / 帮我写个（委婉的）催办 / 帮我写个群通知"。
- 把粗略要点或某条消息，变成可直接发的对外文字。
- ⚠️ 审方案 / 写正式审核意见走 `project-review`（结构更专）；这里是日常沟通文字。

## Procedure

1. **弄清 4 件事**（缺了就问，能从上下文推就别啰嗦地反问）：
   - **给谁**：领导 / 同事 / 合作者 / 用户 / 客户 —— 决定语气与详略。
   - **什么渠道**：飞书消息 / 邮件 / 群通知。
   - **要达成啥**：回复 / 催办 / 通知 / 婉拒 / 道歉 / 确认……
   - **语气**：正式 / 委婉 / 直接 / 简洁 / 温和而坚定。没说就按对象给个稳妥默认。

2. **要回某条具体消息或邮件，先读上下文**：
   - 飞书消息 → 用 `lark-im` 技能搜 / 读相关会话。
   - 邮件 → 用 `lark-mail` 技能读原邮件。
   - 抓住对方真正在问 / 要什么，别答非所问。

3. **出草稿（默认给 2 个版本让用户挑）**：
   - 例如 **A 稳妥版**（周全、客气）+ **B 更直接版**（短、利落）。
   - 按对象调：给领导 **结论先行 + 影响/资源**；给用户 **少术语 + 讲清影响和下一步**；给同事 **简洁可执行**。
   - 涉及 **承诺 deadline / 代表用户表态 / 关键数字** 的地方，**标出来**让用户确认，别替他拍板。

4. **交草稿（默认到此为止，由用户自己发）**：
   - 把 1-2 版草稿发给用户看，并说明两条路：「**复制这版自己发**；或要我**直接发给 X** 我再发。」
   - **默认只给文字、不替用户发**，别自作主张发出去。

5. **只有用户明确说"直接发给 X / 帮我发出去"时，才代发**：
   - 飞书消息 → `lark-im --as user`（以用户身份发，对方看到是用户本人）。**发前再确认一次收件人 + 内容**——外发不可逆。
   - 邮件 → `lark-mail`，优先 **存草稿**让用户最后过目；要发也先确认。
   - ⚠️「预填飞书输入框」做不到（API 不支持），别承诺这个；用户要亲手发就走"复制粘贴自己发"。

## Pitfalls
- **默认只出草稿、由用户自己发**；只有用户明确说"直接发给 X"才代发，代发前必再确认收件人 + 内容。绝不自作主张发出去。
- 「放进飞书输入框」做不到（API 不支持），别承诺这个能力。
- 语气跟着对象走，别一律端着；用户要"简洁"就别堆客套。
- 别擅自替用户承诺时间 / 资源；这类内容标出来让他定。
- 先读懂对方要什么再写，别答非所问。

## Verification
- 用户说"帮我回一下张三问的接口字段" → 先读会话、给 2 版草稿、让用户自己发；用户明确说"直接发给张三"才代发（发前再确认），**默认不**自动发出去。
- 用户说"把这段写正式点" → 给润色版 + 问放哪 / 发不发。
