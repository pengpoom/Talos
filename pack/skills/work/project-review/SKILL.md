## When to Use
- 用户说"帮我审这个方案 / 帮我审一下这文档 / 按我的 SOP 看看有什么问题 / 帮我写审核意见"。
- 把一份方案 / 文档 / 设计稿，按标准挑出问题、给出可发的审核意见。
- ⚠️ 只是"把话写得体 / 发出去"走 `draft-reply`；这里是"按标准挑问题"的结构化审。

## Procedure

1. **拿到并读全要审的东西**：
   - 飞书文档（给了 URL/token）→ 用 `lark-doc` 技能读全文。
   - 用户直接贴的文本 / 要点 → 直接用。
   - 没给内容就先问"审哪份"。

2. **加载审核标准（SOP）**：
   - 读 `~/.hermes/research/review-sop.md`（用户自己的审核清单）；**有就严格按它审**。
   - 没有就用**默认审核镜**：完整性、风险 / 边界情况、可行性、清晰度、与目标一致性；技术方案再加 接口 / 依赖 / 测试 / 回滚 / 兼容。并提醒用户"可把你的 SOP 写进 `review-sop.md`，我下次按你的来"。

3. **审（判断活，要具体）**：
   - 每条问题**指到具体位置 / 原文**，别空泛说"不够清晰"。
   - 分清**硬问题**（必须改）和**建议**（可选优化）。

4. **输出固定 5 段结构**：
   ```
   一、主要问题（指到位置，标 硬/软）
   二、风险判断（影响 + 触发条件）
   三、修改建议（每条给可操作的改法）
   四、可发送审核意见（直接能发的版本）
   五、需要追问的问题（信息不足处）
   ```

5. **发送 / 落库（默认不外发）**：
   - "可发送审核意见"默认**只给草稿**让用户自己发；要发给作者 / 放进文档，走 `draft-reply`（发前预览确认）或 `lark-doc` 追加，**绝不自动外发**。

## Pitfalls
- 问题要**指到具体位置 + 给改法**，别空泛打分。
- 分清"必须改"和"可选优化"，别把建议当硬伤吓人。
- **可发送意见默认只出草稿**；外发 / 改共享文档先预览确认（用户硬底线）。
- 没有 SOP 别瞎编标准——用默认镜 + 提醒用户补 `review-sop.md`。

## Verification
- 用户贴一份方案说"按我 SOP 审" → 读 `review-sop.md`（有则按它）、输出 5 段、问题指到位置，"可发送意见"是草稿不自动发。
- 用户给飞书文档链接说"帮我审" → 用 `lark-doc` 读全 → 同上。
